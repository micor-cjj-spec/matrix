# GL Voucher Dictionary Prompt

## 1. Corresponding BUS Source

- Domain: `fi`
- Subdomain: `gl`
- Biz code: `voucher`
- BUS directory: `docs/bus/fi/gl/voucher/`
- Related BUS docs:
  - `business.md`
  - `fields.md`
  - `flow.md`
  - `rules.md`
  - `page.md`
  - `api.md`

This prompt is used to generate the voucher scene dictionary, field semantics, enums, state definitions, validation semantics, and display semantics for the General Ledger voucher domain.

---

## 2. Generation Objective

Generate a dictionary-oriented document for the voucher scene based on BUS documents and current implementation signals. The output must become the semantic baseline for API, backend, frontend, SQL, and test prompts.

The generated result must cover at least:

1. core business objects
2. voucher header field semantics
3. voucher line field semantics
4. related object semantics
5. enum definitions
6. state definitions and state machine
7. accounting validation semantics
8. amount / currency / precision semantics
9. display semantics and naming consistency
10. pending confirmation items

---

## 3. Input Context

Generation must consume the following context:

### 3.1 BUS business context
Extract from BUS:
- voucher business objective
- voucher lifecycle
- relations with organization, ledger, period, source document, account subject
- approval / posting / reverse / void constraints

### 3.2 BUS field context
Extract from BUS:
- header fields
- line fields
- derived / computed fields
- filter fields
- detail display fields

### 3.3 Current implementation signals
Current implementation history already shows evidence of:
- voucher lines support
- debit/credit balance validation
- GL posting entries generation
- synchronous batch posting
- currency precision and rounding rules
- approval boundary and rejected edit boundary
- reverse linkage display
- date-range list filters

Use these signals as alignment context instead of ignoring current capability.

---

## 4. Generation Constraints

### 4.1 Semantic constraints
- Do not invent business fields or states not supported by BUS or strongly implied by current code.
- If BUS is unclear, mark item as `pending confirmation`.
- Keep one concept mapped to one canonical name.
- Separate business meaning from technical meaning where necessary.

### 4.2 Output structure constraints
The result must contain at least:
1. scene overview
2. core business objects
3. field dictionary
4. enum definitions
5. state machine
6. accounting and validation semantics
7. precision semantics
8. display semantics
9. pending confirmation items

### 4.3 Field dictionary constraints
For each important field, include:
- object name
- field display name
- field code suggestion
- data type suggestion
- required or not
- business meaning
- source / destination
- example value
- notes

### 4.4 Enum and state constraints
Where applicable, define semantics for:
- voucher status
- approval status
- posting status
- reverse relation type
- source type
- debit/credit direction
- import source type

## 7. Acceptance Criteria

The generated result is acceptable only if:
1. it clearly separates voucher header, voucher line, and related object semantics;
2. it covers core states, enums, and validation semantics;
3. it can support API/backend/frontend/SQL/test prompts consistently;
4. it reflects current implementation signals where relevant;
5. it marks unclear BUS items explicitly instead of inventing them.
