# 总账-报表基础平台业务设计（初版）

> 模块：GL（总账）
> 主题：三大报表基础平台（Report Platform）
> 更新时间：2026-03-28

## 1. 业务目标
为资产负债表、利润表、现金流量表提供统一的数据底座、规则配置、计算服务和查询交互能力，避免三张报表分别做一套取数逻辑。

平台建设完成后，应满足以下目标：
- 报表统一基于已过账结果和标准中间层取数
- 报表项目、公式、科目映射尽量配置化
- 前后端对三大报表使用统一查询协议、统一导出协议、统一钻取协议
- 后续支持新增企业自定义报表或监管报表，不必重写核心计算框架

## 2. 建议定位
放在“总账 / 分析报表”能力下，作为三大报表的公共底座。

建议将报表体系分为 4 层：
1. 数据底座层：凭证、总账分录、余额表、现金流补充字段
2. 报表定义层：报表模板、报表项目、公式规则、科目映射
3. 计算服务层：按期间和组织口径进行取数、汇总、校验、钻取
4. 展示交互层：查询页、报表表格、导出、联查、取数说明

## 3. 数据来源与边界
### 3.1 数据来源
- 唯一合法源头：已过账凭证及其生成的总账分录
- 标准查询中间层：余额表
- 现金流补充来源：单据字段、凭证分录现金流项目、现金流调整记录

### 3.2 边界约束
1. 报表页面不允许直接修改报表金额
2. 报表结果应可回溯到余额表、明细账、凭证
3. 报表取数规则尽量配置化，不在页面中写死
4. 非现金流报表优先从余额表取数，不直接扫全部分录
5. 现金流量表允许存在“补录/调整”能力，但需留痕

## 4. 核心业务对象
### 4.1 报表模板（Report Template）
表示一张可查询的报表定义。

建议字段：
- `fid`
- `ftemplate_code` 模板编码，如 `BS-STD`、`PL-STD`、`CF-STD`
- `ftemplate_name` 模板名称
- `freport_type` 报表类型：资产负债表 / 利润表 / 现金流量表
- `fversion` 版本号
- `fstatus` 状态：草稿 / 启用 / 停用
- `forg_scope` 适用组织范围
- `fremark`

### 4.2 报表项目（Report Item）
表示报表中的行定义，是三大报表最核心的业务对象。

建议字段：
- `fid`
- `ftemplate_id`
- `fitem_code` 行项目编码
- `fitem_name` 行项目名称
- `fparent_id`
- `flevel`
- `frow_no` 行次
- `fline_type` 明细行 / 汇总行 / 公式行 / 文本行
- `fsign_rule` 正负方向规则
- `fperiod_mode` 期末 / 年初 / 本期 / 本年累计
- `fdrillable` 是否可钻取
- `feditable_adjustment` 是否允许调整
- `fsort`

### 4.3 取数规则（Report Rule）
定义一个报表项目如何取数，不应分散在 Java 代码 if/else 中。

建议支持的规则类型：
- 按单个科目取数
- 按科目范围取数
- 按科目标签取数
- 按报表项目汇总
- 按公式计算
- 按现金流项目汇总
- 按组织、币种、辅助维度过滤

建议字段：
- `fid`
- `fitem_id`
- `frule_type`
- `frule_expr` 表达式或 JSON 配置
- `fsign_transform`
- `fpriority`

### 4.4 科目报表映射（Account Report Mapping）
用于将会计科目与报表项目建立关系。

当前项目里 `bizfi_fi_account` 已有 `freport_item`、`fpltype`，可作为第一阶段基础，但后续建议独立映射表承接更复杂规则。

建议字段：
- `fid`
- `faccount_id`
- `ftemplate_id`
- `fitem_id`
- `fmapping_type` 直接映射 / 条件映射 / 现金流映射
- `feffective_from`
- `feffective_to`

### 4.5 报表查询快照（Report Snapshot，可选）
用于性能优化和审计追溯。

适用场景：
- 月结后固化报表
- 对外报送版本留档
- 查询性能缓存

## 5. 统一查询口径
### 5.1 公共查询条件
- 组织 / 账套
- 会计期间
- 币种
- 是否包含未记账调整
- 是否显示零值行
- 模板版本

### 5.2 公共输出结构
建议所有报表接口统一输出：
- 报表头：组织、期间、币种、模板、生成时间
- 报表体：行项目列表
- 校验区：平衡校验、勾稽校验、异常提示
- 钻取信息：每行是否可下钻、下钻目标类型

### 5.3 报表行输出建议
- `itemCode`
- `itemName`
- `rowNo`
- `values`
- `displayStyle`
- `drillable`
- `warnings`
- `sourceSummary`

## 6. 与现有模块关系
### 6.1 与凭证模块关系
- 报表只统计 `POSTED` 凭证产生的数据
- 冲销凭证按正常负向参与计算
- 凭证反过账或异常修复后需触发报表重算或缓存失效

### 6.2 与余额表关系
- 资产负债表、利润表优先从余额表读取
- 余额表是三大报表的统一财务事实层
- 报表计算服务不应绕过余额表直接拼凑口径

### 6.3 与科目主数据关系
- 科目上已有 `freport_item` 可作为第一阶段映射来源
- 科目上已有 `fpltype` 可作为利润表分类辅助字段
- 科目上已有 `fcash`、`fbank`、`fequivalent` 可辅助现金流量口径识别

## 7. 后端能力建议
建议在 `fi-service` 下新增或归整以下能力：
- 报表模板管理
- 报表项目管理
- 报表规则管理
- 报表查询服务
- 报表导出服务
- 报表校验服务
- 报表钻取服务

建议接口分层：
- `GET /report/templates`
- `GET /report/templates/{id}`
- `GET /report/balance-sheet`
- `GET /report/profit-statement`
- `GET /report/cash-flow`
- `GET /report/{type}/drilldown`
- `POST /report/template`
- `PUT /report/template/{id}`

## 8. 前端能力建议
在 `matrix-web` 中，建议统一做一套报表查询页框架，再衍生三张报表页面。

统一能力：
- 顶部查询栏
- 主报表表格
- 校验结果提示
- 行项目钻取
- Excel 导出
- 取数说明抽屉

统一交互规则：
1. 查询条件变化后手动点击“查询”
2. 允许保存常用查询方案
3. 报表行点击后打开下钻视图，不直接跳走主页面
4. 异常行显示来源说明和修复建议

## 9. 权限与审计
建议拆分权限：
- 报表查询
- 报表模板维护
- 报表取数规则维护
- 报表调整
- 报表导出

审计要求：
- 规则变更留痕
- 模板启停留痕
- 现金流补录/调整留痕
- 已出具报表的导出记录留痕

## 10. 分期落地建议
### Phase 1
- 建立报表模板、报表项目、报表规则基础表结构
- 完成统一查询 DTO / VO
- 前端完成统一报表页面骨架
- 先支持资产负债表、利润表

### Phase 2
- 完善钻取、导出、公式配置
- 接入校验规则
- 完善报表模板维护页

### Phase 3
- 接入现金流补录与调整
- 引入报表快照
- 支持多版本模板和企业个性化模板
